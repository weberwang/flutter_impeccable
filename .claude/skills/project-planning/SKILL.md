# Project Planning For Executable Product Delivery

## Purpose

- 把用户的需求目标、产品想法、技术栈讨论和产品方向分歧，收敛成一份可以执行落地的产品规划，而不是停留在泛化建议。
- **需求必须基于事实**：优先验证需求是否真实存在（用户反馈、市场调研、已验证假设），如果需求不成立或缺乏基础，直接返回结论而不继续规划。
- 在讨论过程中识别关键决策点，明确哪些必须先定，哪些可以延后，哪些需要用分支方案并行评估。
- 默认优先输出模块设计文档；根据任务需要再补完整规划、阶段路线图、分支比较结论或设计分支建议。

## When To Use

- 需要把一个模糊需求或产品想法整理成可执行的产品方案。
- 需要围绕用户目标、目标用户、场景、约束和商业目标，形成一份能继续推进研发或设计的计划。
- 需要讨论技术栈、架构方向、平台范围、MVP 范围、发布节奏或迭代顺序。
- 需要把 PRD 或产品描述拆成独立模块，并为每个模块生成单独的设计说明。
- 需要比较多个产品分支，例如轻量 MVP、平台优先、增长优先、企业版优先或重构优先。
- 需要比较多个设计分支，例如保守迭代、体验优先、品牌优先、效率优先或不同信息架构方案。

## Do Not Use

- 纯代码实现任务。
- 纯框架 API 问答。
- 不需要收敛方案、只想随意头脑风暴的闲聊。
- 已经有明确实施方案且只缺单一文案或单一页面改稿的任务。

## Default Outputs

根据用户目标和需求验证结果，优先产出以下一种或多种：

### 需求未通过验证时

- 一份需求验证结论：当前信息、缺口、建议验证方式、预估周期、何时可重启规划。

### 需求通过验证时

- 一份模块设计包：模块目标、用户流、核心能力、边界、数据输入输出、依赖、验收标准。
- 一份可执行产品规划：目标、范围、约束、阶段、优先级、验收口径、风险与下一步。
- 一份分支比较：候选方向、前提假设、取舍、成本、风险、适用阶段、推荐结论。
- 一份设计分支建议：以视觉方向为主，包含品牌气质、信息层级、内容密度、交互表达、实现成本和推荐路径。
- 一份阶段路线图：MVP、Beta、正式版、增长阶段的功能顺序与里程碑。

默认假设：

- 若用户未提供需求来源或验证信息，优先追问而不是假设需求成立。
- 若用户未指定输出类型，先产出模块设计文档（前提是需求已通过验证）。
- 若用户未指定阶段，按正式版目标倒推范围、模块和里程碑。
- 若用户提到设计分支但未指定深度，优先讨论视觉方向，不默认下钻到页面级细节。

## Working Principles
### 0. Requirements Must Be Grounded In Facts

- 规划之前必须验证需求是否真实存在，不能基于假设或猜测开始产品设计。
- 需求验证信息源：用户反馈、市场调研数据、已验证的商业假设、现有数据或已上线产品反馈。
- 如果需求缺乏事实支撑、用户验证不足、市场信号不清或前置假设不成立，**直接返回验证结论，不继续制定产品规划**。
- 这不是"否定"，而是"暂停"：明确缺口、下一步如何获取证据、何时可以重启规划。
### 1. Planning Must End In Execution

- 任何讨论都要落到可执行项，至少给出阶段目标、负责人视角的任务块、依赖和进入下一步所需信息。
- 不把“可以考虑”当结论；如果信息不足，要明确缺口和补齐方式。

### 2. Separate Facts, Assumptions, And Decisions

- 已知事实、待验证假设和已经决定的事项必须分开写。
- 如果结论依赖未验证前提，要明确标注该前提。

### 3. Use Branches Only For Real Decisions

- 只有当方向差异会显著影响产品范围、技术方案、时间成本、体验或商业目标时，才建立分支比较。
- 分支不应只是换个说法；每个分支都必须有明确的战略偏向和适用条件。

### 4. Decompose By User Value And Delivery Boundaries

- 模块拆分优先围绕用户价值链路、业务能力边界、数据边界和交付边界，而不是按页面数量机械拆分。
- 设计分支优先围绕视觉方向、品牌气质、信息层级、核心操作路径和实现约束，而不是只比较配色或风格词。

## Required Inputs

能直接开始时不必一次性问全，但至少要尽快补齐：

- 目标产品或功能想解决什么问题。
- 目标用户是谁，核心场景是什么。
- 当前阶段是什么，例如从 0 到 1、验证期、MVP、增长期、重构期。
- 时间、团队、预算、平台或合规限制。
- 是否已有既定技术栈、设计系统、品牌方向或现有资产。
- 当前最需要的输出是什么，例如完整规划、模块拆解、路线图或分支比较。

## Clarification Rules

### Requirement Validation First

如果用户提出产品想法或需求，首先验证需求基础，而不是直接开始规划：

- **问需求来源**：这个需求来自用户反馈、市场调研还是内部想法？已经验证过用户是否真的需要吗？
- **问用户信号**：是否有具体用户反馈、已上线产品数据或明确的商业假设可以支撑？
- **问前置假设**：规划依赖的核心假设是什么？这些假设是验证过的吗？
- **决策点**：如果需求缺乏事实支撑，立即返回"需求待验证"结论，指出缺口和后续验证方式，不继续规划。

### Planning-Related Clarifications

如果需求通过验证，再针对规划本身追问最少量的高价值问题：

- 若目标不清，先问目标用户、核心问题和成功标准。
- 若阶段不清，先问当前是验证需求还是准备交付。
- 若范围不清，先问必须上线的最小能力和明确排除项。
- 若技术栈在讨论中，先问已有约束、团队能力和非功能要求。
- 若要拆模块，先问主流程、关键对象和系统边界。
- 若要做设计分支，先问信息密度、品牌倾向、平台形态和实现约束。

## Planning Flow

### 0. Validate Requirement Foundation (Gate)

所有规划的第一步：验证需求是否有事实基础。

- 收集需求信息源：用户反馈数量、市场调研结果、已验证的商业假设、现有产品数据或竞品分析。
- 判断需求成立度：是否有足够的用户信号表明这个需求真实存在？是否有市场验证？
- 决策：
  - **通过**：需求有事实支撑，继续下一步规划。
  - **待验证**：需求基础不足，直接输出"需求待验证"结论，明确缺口、建议验证方式、估计验证周期，停止规划。
  - **不成立**：基于现有信息判断需求不成立或优先级低，直接输出结论和原因，不启动规划。

### 1. Frame The Planning Target

- 识别当前任务属于哪一种：完整产品规划、模块拆解、技术栈取舍、产品分支比较、设计分支比较、路线图重排。
- 定义本轮输出的完成标准，例如“得到一版可执行 MVP 方案”或“得到支付模块设计文档”。
- 明确该任务的时间视角：本周可落地、一个版本内落地、季度规划或长期方向；未指定时按正式版视角组织。

### 2. Extract The Product Core

- 归纳目标用户、核心问题、使用场景、关键动作和成功结果。
- 用一到三句话写清产品价值主张。
- 标记确定项、争议项和未知项。

### 3. Set Delivery Constraints

- 明确平台范围、团队规模、预算、上线窗口、监管要求、性能要求和已有系统约束。
- 判断哪些约束是硬约束，哪些只是偏好。

### 4. Choose The Planning Path

#### Path A: Executable Product Plan

- 定义目标、目标用户、核心场景、MVP 边界和非目标。
- 生成功能优先级，区分必须、有利但可延后、明确不做。
- 给出阶段划分、里程碑、关键依赖、主要风险和验证方式。

#### Path B: Product Branch Comparison

- 列出 2 到 4 个真实分支，例如“轻量验证版”“平台能力优先版”“增长闭环优先版”。
- 对每个分支分析目标、关键能力、代价、风险、适用阶段和放弃项。
- 给出推荐分支，并说明为什么不是其他分支。

#### Path C: Module Design Decomposition

- 先找主流程，再按用户价值链路和系统边界拆模块。
- 为每个模块定义：目标、用户任务、入口、输出、依赖、状态、边界、验收标准。
- **必须设计项目 Main 模块**：用于串联其他模块、组织主线导航的总模块，通常以主页的形式承载，负责承接用户进入产品后的主页面、展示核心入口、组织模块切换，并串联关键任务路径。Main 模块承担系统主组织层的作用，决定用户如何进入、理解和使用各个模块。
- 标出可并行推进模块与必须前置模块。
- 若用户没有要求完整 PRD，默认先把讨论收敛为模块设计包，而不是铺开整份大而全规划。

#### Path D: Design Branch Discussion

- 识别设计变量：视觉方向、品牌表达、信息层级、内容密度、交互深度和实现复杂度。
- 为每个设计分支说明适用场景、品牌感受、用户收益、实现代价和维护成本。
- 输出推荐视觉方向，并说明需要保留的备选分支和触发切换条件。

#### Path E: Tech Stack And Delivery Alignment

- 技术栈讨论必须回到产品目标、团队能力、迭代速度、维护成本和关键风险。
- 不只比较技术优劣，要比较对产品分期、模块边界、组织协作和设计能力的影响。
- 若无法定案，输出“当前建议 + 决策前必须验证项”。

### 5. Convert Discussion Into Deliverables

根据任务输出对应文档结构：

#### Executable Product Plan

- 背景与目标
- 目标用户与场景
- MVP 范围与排除项
- 功能优先级
- 阶段计划与里程碑
- 风险、依赖、验证方式
- 下一步行动

#### Module Design Doc

- 模块目标
- 用户流或业务流
- 核心能力
- 数据与状态边界
- 依赖与接口假设
- 正式版约束下的范围取舍
- 设计要点
- 验收标准

**Main 模块特殊要求**：
- 定义自身职责范围：作为主页或主页面，承接用户进入产品后的第一层体验，提供核心导航、模块入口和关键主线任务的统一承接。
- 明确各模块如何通过 Main 被用户感知和进入，包括首页一级入口、二级路径和跨模块主线的组织方式。
- 界定哪些功能必须由 Main 模块统一承接，哪些保留在功能模块内部完成。

#### Branch Comparison

- 分支列表
- 每个分支的核心取向
- 收益、成本、风险、适用条件
- 推荐结论
- 不采纳分支的原因

#### Design Branch Recommendation

- 分支名称与定位
- 视觉与品牌表达差异
- 信息层级与内容密度差异
- 交互复杂度差异
- 实现与维护成本
- 推荐方案与保留条件

### 6. Add Decision Discipline

- 每个关键结论都要标注：已定、待确认、需验证。
- 若存在阻塞，明确是谁需要提供什么信息，才能推进到下一步。
- 若需要后续跨职能协作，写清产品、设计、研发各自的后续动作。

## Branch Heuristics

### Prefer MVP Branch When

- 目标是快速验证需求、用户和付费意愿。
- 团队资源有限，且上线窗口紧。
- 当前最大的风险是“是否有人要”，不是“能不能做出来”。

### Prefer Platform Or Architecture Branch When

- 后续会快速扩模块、扩团队或扩端。
- 现有系统会显著拖慢后续交付。
- 非功能要求会直接决定产品能否成立。

### Prefer Growth Branch When

- 核心价值已经成立，当前瓶颈是转化、留存或传播。
- 数据闭环、激励机制、分享链路或运营后台成为关键。

### Prefer Design Branch Exploration When

- 核心问题不在功能缺失，而在认知成本、信息组织、品牌表达或视觉气质。
- 同一功能在不同用户心智下需要不同视觉方向或信息呈现方式。

## Quality Bar

完成的规划结果应满足：

- 目标清晰：能说明为谁解决什么问题。
- 范围清晰：能分辨必须做、可以后做和不做。
- 决策清晰：关键分支和推荐结论明确。
- 可执行：下一步任务块、依赖和里程碑明确。
- 可协作：产品、设计、研发都知道自己下一步要做什么。
- 可验证：存在可衡量结果、验收标准或验证路径。

## Output Style

- 优先输出结构化文档，不输出空泛长文。
- 必要时先给一版骨架，再补每个模块或分支细节。
- 如果输入信息不足，先补关键问题，不伪装成确定结论。
- 如果用户已经给出强约束，围绕约束优化，而不是重新发散。
