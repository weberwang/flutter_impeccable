# Flutter Stack Selection

## Purpose

- 基于产品设计方案，选出最合适的 Flutter 技术方案，而不是泛化跨技术栈推荐。
- 确保 Flutter 方案与配套包/插件形成最佳实践组合，兼顾开发效率、可维护性、性能与可扩展性。
- 输出可执行的技术决策结果：推荐 Flutter 方案、备选方案、插件搭配、落地步骤、风险控制与验证计划。

## Flutter Scope Rule

- 默认只讨论 Flutter 生态与 Flutter 项目常见后端/平台配套，不扩展到非 Flutter 前端主栈对比。
- 包/插件必须按“推荐方案”成套给出，不做孤立的单插件罗列。
- 每个插件建议都要说明其在推荐方案中的职责、依赖关系与替代选项。
- [根据需求采用推荐方案，插件采用推荐方案搭配]

## Scope Defaults

- 默认作用域：工作区级（团队共享）。
- 默认深度：完整评估版（含评分、风险、路线图）。
- 若用户明确要求快速结论，则切换为快速清单版，并在末尾附“待补充验证项”。

## When To Use

- 用户已有产品设计方案，需要落地为 Flutter 技术方案。
- 需要为 Flutter 应用做架构决策，并给出配套包/插件组合。
- 需要把设计目标转换为 Flutter 端能力、后端配套与工程化约束。
- 需要给出 Flutter 项目从 MVP 到正式版的演进路线，而非一次性重架构。

## Do Not Use

- 纯 API 语法问答。
- 已有明确栈且只做小范围代码修复。
- 与产品目标无关的“流行框架罗列”。
- 非 Flutter 主栈对比（如 React/Vue/Angular 前端选型）作为主输出。

## Required Inputs

最少需要以下信息才能可靠决策：

- 产品类型与核心流程：内容型、交易型、协作型、工具型等。
- Flutter 目标平台：iOS、Android、Web、Desktop（可多选）。
- 阶段目标：MVP 验证、正式版交付、增长扩展、重构。
- 非功能要求：性能、并发、稳定性、安全合规、国际化、多租户。
- 团队约束：人数、经验栈、交付周期、预算、运维能力。
- 设计方案特征：交互复杂度、动画密度、离线需求、实时能力、数据规模。
- Flutter 工程偏好：状态管理、路由方案、本地存储、网络层、代码生成策略。

## Clarification Rules

如果信息不足，优先追问高价值问题：

1. 若业务目标不清：先问核心用户、关键价值和成功指标。
2. 若平台不清：先问首发平台与未来扩展平台。
3. 若阶段不清：先问当前重点是验证速度还是长期可扩展。
4. 若团队能力不清：先问现有技术栈熟悉度与可招聘性。
5. 若约束不清：先问合规、数据驻留、预算、上线窗口。

## Decision Flow

### 1. Translate Design Into Technical Drivers

把设计方案转译为技术驱动项：

- 渲染与交互复杂度：影响 Flutter 渲染策略、状态管理和组件分层。
- 多端一致性要求：影响 Flutter 多平台适配策略与平台差异处理。
- 实时与协作需求：影响后端通信协议、缓存与事件架构。
- 数据关系复杂度：影响本地存储方案、远端数据建模与同步策略。
- 版本迭代速度：影响 Flutter 工程模块化、代码生成与发布流程设计。

### 2. Establish Architecture Boundary First

先定边界，再选技术：

- 客户端层：Flutter UI 体系、状态管理、路由、网络层、错误处理。
- 服务层：BFF、业务服务、异步任务、实时通道（按 Flutter 客户端需求对齐）。
- 数据层：主存储、缓存、搜索、对象存储。
- 平台层：CI/CD、发布渠道、崩溃监控、日志、告警。

### 3. Build Candidate Stack Combinations

每个候选组合必须覆盖端到端，而不是单点技术：

- Flutter 客户端方案 + 服务框架 + 数据库 + 认证 + 发布部署 + 观测。
- 标注“推荐搭配关系”，例如：
  - 状态管理套件（状态管理 + 路由 + 依赖注入 + 代码生成）。
  - 数据访问套件（网络层 + 序列化 + 缓存 + 本地数据库）。
  - 工程化套件（Lint + 测试 + CI + 崩溃监控 + 性能分析）。

### 4. Evaluate By Weighted Criteria

使用加权评分，避免主观拍脑袋：

- 交付速度
- 团队匹配度
- 长期可维护性
- 性能与成本效率
- 生态成熟度
- 安全与合规
- 可观测性与故障恢复能力

默认建议权重：

- MVP：速度 35%，团队匹配 25%，维护性 15%，其余 25%。
- 正式版：维护性 25%，性能成本 20%，安全合规 20%，其余 35%。

### 5. Decide With Branch Logic

- 如果目标是快速验证且团队偏前端：优先全栈同构或托管后端组合。
- 如果存在高并发和复杂业务边界：优先分层明确、可独立扩展的服务化组合。
- 如果多端体验一致性与交互复杂度高：优先强 UI 能力跨端方案并配套统一设计系统。
- 如果合规与数据治理要求高：优先可控部署与可审计架构，减少强绑定托管能力。

### 6. Recommend Bundled Ecosystem

为每个核心技术补充搭配的高效工具与插件：

- Flutter 核心工具链：代码生成、测试、调试、热重载、格式化。
- Flutter 包与插件：状态管理、路由、网络、本地存储、权限、推送、分析。
- 集成能力：与后端、认证、支付、地图、消息系统的官方或高质量 SDK。
- 运维观测：崩溃监控、性能分析、日志追踪、埋点分析相关插件。
- 团队协作：代码规范、文档生成、版本发布、变更追踪工具。

目标：让团队开盒即用，减少自主选型和集成成本。

插件搭配规则：

- 先给“推荐方案”的主插件组合，再给兼容替代组合。
- 避免功能重叠插件并存；若并存必须说明边界。
- 每个插件标注阶段优先级：MVP 必备 / 正式版可选 / 增长阶段考虑。

### 7. Produce Final Recommendation

最终必须同时输出：

- 主推荐组合：当前阶段最优，含完整生态配套。
- 次推荐组合：在不同约束变化下可切换。
- 不推荐组合：说明为什么不适合当前产品。

## Best-Practice Compatibility Checks

推荐前必须通过以下检查：

1. 前后端契约一致性：接口定义、错误码、版本策略是否统一。
2. 身份认证与权限模型：是否支持最小权限、会话安全与审计。
3. 数据一致性与扩展策略：事务边界、缓存失效、读写扩展路径。
4. 可观测性：日志、指标、追踪是否可贯通到核心用户路径。
5. 部署与回滚：是否支持蓝绿/金丝雀或等价安全发布机制。
6. 本地开发体验：一键启动、环境一致、CI 可复现。
7. 测试策略：单测、集成、端到端覆盖关键链路。
8. 工具链与插件生态：配套工具是否官方维护、社区活跃度是否足够、集成难度是否合理。

## Output Format

默认输出以下结构：

1. 设计目标摘要与关键技术驱动项
2. 候选技术栈组合（2-4 套）
3. 加权评分表与结论
4. 主推荐方案（含完整技术搭配）
5. 配套工具与插件清单（按推荐方案成套输出）
  - Flutter 应用层（状态管理、路由、UI、国际化）
  - Flutter 数据层（网络、序列化、本地存储、缓存）
  - Flutter 设备能力层（权限、通知、相机、定位、文件）
  - 工程与运维层（测试、CI/CD、崩溃监控、日志、性能分析）
  - 每项标注：职责、依赖、替代、阶段优先级
6. 风险与规避策略
7. 90 天落地路线图（MVP -> 正式版）
8. 决策复审触发条件

## Output Modes

### Mode A: 快速清单版

- 适用于时间紧、先要方向结论的场景。
- 输出：主推荐栈、1 个备选栈、3 个关键风险、2 周行动清单。
- 限制：必须显式标注“未完成的验证项”。

### Mode B: 完整评估版（默认）

- 适用于正式技术决策与跨角色评审。
- 输出：候选栈比较、加权评分、主备方案、迁移/演进路径、90 天路线图。
- 限制：必须包含“为什么当前不选其他方案”。

## Completion Criteria

完成标准：

- 有且仅有一个当前主推荐组合。
- 至少一个可落地备选组合。
- 每个组合都覆盖 Flutter 客户端、后端配套、数据、认证、发布、观测。
- 明确说明技术之间为什么“搭配合理”。
- 每个核心模块都有配套工具与插件清单，包含官方推荐或最佳实践方案。
- 配套工具必须标注优先级：MVP 必备 / 正式版可选 / 增长阶段考虑。
- 给出可执行的近期实施顺序，而非概念描述。
- 结论必须可复审：包含触发复盘的时间点或指标阈值。
