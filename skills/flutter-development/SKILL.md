# Flutter 项目开发落地模板

## 核心定位

- 这是一个仓库专属的实施型 skill，用来把任务直接落到当前 Flutter 项目代码中。
- 目标不是输出通用建议，而是用最小改动满足当前项目的架构、规范和交付要求。
- 在使用前，先补充本项目的技术栈、目录结构、约束和生成链路说明。

## 何时使用

- 需要在当前仓库实现或修改功能。
- 需要修复页面、状态、路由、接口、数据链路或模型问题。
- 需要判断代码应放在哪一层，或应该新增哪一种工件。
- 需要把需求约束到当前项目既有模式，而不是临时发明新写法。

## 不适用场景

- 只做设计评审，不改代码。
- 只想写说明文档、流程文档或产品方案。
- 只是通用 Flutter 问答，与当前仓库无关。
- 任务没有代码锚点，且目标模块、流程归属、后端契约都不清楚。

## 默认目标

- 先判断任务落在哪一层。
- 再按该层的固定模式实施。
- 只做完成任务所需的最小修改。
- 变更若涉及项目生成体系、资源同步或脚本流程，按当前项目既有方式处理。

## 实施流程

### 1. 识别任务落点

- 先确定目标模块。
- 再判断它属于 UI、State、Domain、Data、API、Model 还是 Route。
- 如果一个需求跨多层，按当前项目的数据流顺序展开，而不是从页面直接连到底层。

### 2. 定位最小代码锚点

- 优先找直接控制行为的页面、组件、状态定义、业务编排、数据访问、接口契约、模型、映射、路由文件。
- 只改与任务直接相关的文件，不顺手重构无关代码。

### 3. 套用该层的固定模式

- UI 任务优先复用现有页面结构、状态组件、主题样式和交互模式。
- 数据任务遵循当前项目既有的数据访问链路。
- 模型、状态或接口任务优先走当前项目约定的声明和生成方式。

### 4. 同步相邻影响面

- 用户可见文案变化时，同步检查国际化资源。
- 页面跳转方式变化时，同步检查命名路由、导航封装或节流逻辑。
- 后端字段、状态码、枚举或响应结构变化时，同步检查映射、错误转换和兼容分支。
- 注解源文件、资源定义或脚本输入变化时，同步刷新相关产物。

### 5. 完成检查

- 文件是否放在正确层级。
- 是否仍然遵守当前项目的架构、状态管理、导航、数据链路和生成约束。
- 是否引入了项目禁止的实现模式。
- 是否遗漏文案、路由、映射、资源或生成产物同步。

## 决策分支

- 如果任务只是页面视觉调整，但不涉及业务状态，优先改页面或组件，不额外上升到业务层。
- 如果任务位于显示层，但本质只是接线、生命周期修复、空指针修复或局部状态整理，先直接实施，不额外扩展方案。
- 如果任务新增异步动作、提交流程或可恢复状态，优先放到当前项目约定的状态层或业务层。
- 如果任务新增后端接口或字段，优先补齐项目规定的数据链路，再让 UI 接入。
- 如果任务只是表单临时输入、焦点、滚动或动画控制，优先留在页面本地状态。
- 如果任务需要新增模型、状态类或 DTO，优先走当前项目约定的模型定义和生成方案。
- 如果任务本质是设计评审、流程批评或改稿拆单，优先使用对应的设计类 skill，而不是本 skill。

## 输出偏好

- 默认直接实施代码修改，而不是长篇解释。
- 总结时只保留实施摘要、关键修改点、是否涉及生成链路、是否存在阻塞。
- 如果信息不足，只追问真正阻塞落地的点，例如目标模块、页面归属、主动作或后端契约。

## 完成标准

- 代码满足当前任务，不额外扩展功能。
- 改动遵循当前项目目录和职责边界。
- 没有引入项目明令禁止的实现模式。
- 需要同步的生成物、资源或配置已处理。
- 文案、导航、映射、状态分支等相邻依赖已同步。
